基于SDD规格驱动开发流程完整演示个人记账本的开发过程。

规格驱动开发的核心流程是五步：需求收集 \-\> 功能规格 \-\> 技术设计 \-\> AI评审 \-\> AI生成代码。

本案例的流程：需求收集 \-\> 功能规格 \-\> 技术设计 \-\> AI评审 \-\> AI生成代码  \-\> AI辅助验证 \-\> 人工联调测试

\-\> 项目部署 \-\> 集成测试 \-\> 问题修复和功能完善 \-\> 交付

职场新人、学生、自由职业者，希望快速记录日常收支并通过报表了解消费结构。具体需求详见：5\-1\-1 个人记账本项目需求搜集

**## 7. 认证与安全**

**### 7.1 JWT 流程**

1. 用户登录/注册成功后，后端返回 JWT Token
2. 前端将 Token 存入 `localStorage`，并在每次请求时通过 `Authorization: Bearer <token>` 携带
3. 后端通过中间件验证 Token 有效性，解析出 `user_id`
4. Token 有效期：7 天

**### 7.2 密码安全**

- 使用 `passlib` + `bcrypt` 对密码进行哈希存储
- 不存储明文密码

**### 7.3 验证码安全**

- 验证码有效期 5 分钟
- 验证码使用后立即标记为已使用
- 60 秒内不可重复发送（基于邮箱的冷却机制）
- 开发环境下，验证码固定为 `123456` 方便调试

**### 7.4 数据隔离**

- 所有数据查询均携带 `user_id` 过滤条件
- API 层通过中间件从 JWT 中解析 `user_id`，业务层强制使用

**## 8. 错误码定义**

| 错误码 | 说明 |
|--------|------|
| 0 | 成功 |
| 40001 | 验证码错误或已过期 |
| 40002 | 邮箱已注册 |
| 40003 | 邮箱格式不合法 |
| 40004 | 密码长度不足 6 位 |
| 40005 | 金额必须大于 0 |
| 40006 | 金额超出范围 |
| 40007 | 分类名称已存在 |
| 40008 | 备注不能超过 200 字 |
| 40009 | 验证码发送过于频繁 |
| 40010 | 分类下有账单，确认删除 |
| 40101 | Token 无效或已过期 |
| 40102 | 未登录 |
| 40401 | 资源不存在 |
| 50001 | 服务器内部错误 |
